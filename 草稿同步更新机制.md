# 草稿同步更新机制

## 📋 概述

草稿管理系统支持智能同步更新，可以自动获取 Karakeep 中书签的最新变化，同时保留本地的管理状态。

## 🔄 同步逻辑

### 1. 新增草稿

当 Karakeep 中有新书签时：

```typescript
// 首次同步
Karakeep 书签 → 创建本地草稿（status=pending）
```

**处理**：
- 创建新记录
- 自动生成分类建议
- 检测重复
- 状态默认为 `pending`

### 2. 更新草稿

当 Karakeep 中书签内容更新时（例如 AI 处理完成）：

```typescript
// 再次同步
Karakeep 书签（已更新）→ 更新本地草稿
```

**更新字段**（来自 Karakeep）：
- ✅ `title` - 标题
- ✅ `url` - 链接
- ✅ `description` - 描述
- ✅ `summary` - AI 总结
- ✅ `note` - 笔记
- ✅ `favicon_url` - 图标
- ✅ `image_url` - 图片
- ✅ `tagging_status` - 标注状态
- ✅ `summarization_status` - 总结状态
- ✅ `tags_suggestion` - 标签建议
- ✅ `karakeep_created_at` - 创建时间
- ✅ `karakeep_updated_at` - 更新时间
- ✅ `content` - 周刊格式内容
- ✅ `slug` - URL 标识
- ✅ `source` - 来源
- ✅ `word_count` - 字数

**保留字段**（本地管理）：
- 🔒 `status` - 草稿状态（pending/adopted/rejected）
- 🔒 `priority` - 优先级
- 🔒 `category_suggestion` - 分类建议（首次生成后不变）
- 🔒 `content_id` - 关联的内容 ID
- 🔒 `duplicate_of_draft_id` - 重复标记

### 3. 跳过已处理的草稿

当草稿已被采用或拒绝时：

```typescript
// 双向同步
本地草稿（adopted/rejected）→ Karakeep 归档（archived=true）
                                    ↓
                           下次同步时自动跳过
```

## 🎯 使用场景

### 场景 1：AI 处理延迟

```
时间轴：
T1: 收藏书签 → Karakeep（AI 处理中）
T2: 第一次同步 → 被过滤（summarizationStatus != success）
T3: AI 处理完成 → Karakeep（AI 处理完成）
T4: 第二次同步 → ✅ 成功同步，包含 AI 总结和标签
```

### 场景 2：内容更新

```
时间轴：
T1: 同步书签 → 本地草稿（status=pending）
T2: 在 Karakeep 中添加笔记
T3: 再次同步 → ✅ 更新笔记，状态仍为 pending
T4: 采用草稿 → status=adopted，Karakeep archived=true
```

### 场景 3：标签优化

```
时间轴：
T1: 同步书签 → 本地草稿（AI 标签：3 个）
T2: Karakeep AI 重新处理 → 优化标签（5 个）
T3: 再次同步 → ✅ 更新标签建议，保留本地分类和优先级
```

## 📊 同步统计

每次同步会显示详细统计：

```
同步完成（共 50 条）：
- 新增 5 条
- 更新 10 条
- 未变化 30 条
- 去重 3 条
- 失败 2 条
```

**统计说明**：
- **新增**：Karakeep 中的新书签
- **更新**：内容有变化的已存在书签
- **未变化**：内容完全相同，跳过更新
- **去重**：检测到的重复书签
- **失败**：处理失败的书签（如 AI 未完成）

## 🔍 智能更新判断

系统会检查以下字段是否变化：

```typescript
const hasChanges = 
  existing.title !== new.title ||
  existing.summary !== new.summary ||
  existing.description !== new.description ||
  existing.note !== new.note ||
  existing.tagging_status !== new.tagging_status ||
  existing.summarization_status !== new.summarization_status;
```

**优化逻辑**：
- 有变化 → 立即更新
- 无变化 且 status != pending → 跳过（节省时间）
- 无变化 且 status == pending → 更新 `synced_at`（保持活跃）

## 💡 最佳实践

### 1. 定期同步

**推荐频率**：
- 活跃期：每 30 分钟同步一次
- 日常：每 2-4 小时同步一次
- 夜间：每天凌晨同步一次

### 2. 处理流程

```
1. 收藏书签到 Karakeep
2. 等待 5-10 分钟（AI 处理）
3. 执行同步（自动获取 AI 结果）
4. 审核草稿，设置优先级
5. 采用/拒绝（自动归档到 Karakeep）
6. 下次同步自动跳过已处理的
```

### 3. 手动触发更新

如果需要强制更新某个草稿：

**方法 1：等待自动同步**
- 点击"同步"按钮
- 系统自动更新所有 pending 状态的草稿

**方法 2：删除后重新同步**
- 删除本地草稿
- 点击"同步"按钮
- 重新创建（包含最新数据）

## 🚨 注意事项

### 保护机制

1. **状态保护**：
   - adopted/rejected 状态的草稿不会重新同步
   - 这些草稿已在 Karakeep 中归档

2. **优先级保护**：
   - 手动设置的优先级不会被覆盖
   - 分类建议只在首次创建时生成

3. **错误容忍**：
   - 单条记录失败不影响其他记录
   - 归档失败不阻塞本地更新

### 数据一致性

```
Karakeep              本地草稿
  ├─ 书签内容  ────────► Karakeep 相关字段（自动更新）
  │
  └─ archived    ◄──── 本地状态字段（手动管理）
                 │
                 └─ adopted/rejected 时自动归档
```

## 🔧 故障排查

### 问题：草稿没有更新

**检查清单**：
1. ✅ 草稿状态是否为 `pending`？
2. ✅ Karakeep 中内容是否真的有变化？
3. ✅ 查看控制台日志是否显示"更新"？
4. ✅ AI 处理状态是否为 `success`？

### 问题：已采用的草稿又出现了

**可能原因**：
1. ❌ 归档失败（Karakeep API 错误）
2. ❌ 网络问题导致归档请求未发送

**解决方案**：
1. 检查后端日志
2. 手动在 Karakeep 中归档该书签
3. 再次同步时会自动跳过

## 📈 性能优化

### 同步效率

| 场景 | 首次同步 | 后续同步 | 优化比例 |
|------|---------|---------|---------|
| 500 条书签 | 5 页 × 30s | 1 页 × 6s | 80% ⬆️ |
| 100 条书签 | 1 页 × 10s | 1 页 × 3s | 70% ⬆️ |

### 优化策略

1. **参数优化**：
   - `limit=100` - 最大化单次请求
   - `archived=false` - 跳过已归档

2. **智能过滤**：
   - AI 未完成 → 跳过
   - 内容无变化 → 跳过

3. **双向同步**：
   - 已处理 → 自动归档
   - 下次同步 → 自动跳过

## 🎓 示例代码

### 手动触发更新

```typescript
// 在前端
import { useSyncDrafts } from '@/hooks/queries';

const { mutateAsync } = useSyncDrafts();

// 执行同步
const stats = await mutateAsync();
console.log(`更新了 ${stats.updated} 条草稿`);
```
